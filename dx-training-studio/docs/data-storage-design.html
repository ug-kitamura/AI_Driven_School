コンテンツ構造 — フォルダが「正」

レッスンの本文は個別の .md ファイルとして保存します。フォルダの階層構造そのものがシリーズ・コース・レッスンの関係を表すので、フロントマターが壊れていても構造を読み取れます。

_series-order.json ← シリーズ表示順のみ

フォルダ名でシリーズを識別し、UIでの表示順序だけをここに記録する。シリーズ名はフォルダ名から自動取得するので、アプリ名・アイコンはコードに固定。

["series-git", "series-python", "series-tutorial"]
_course.json ← コースメタデータ

コースの受講対象・前提コース・次のコースを記録。曼陀羅グラフのエッジ情報にもなる。コースフォルダ内に1ファイル。

{"target_audience": "Git未経験者", "prerequisites": [], "next_courses": ["..."]}
01_レッスン名.md ← レッスン本文

数値プレフィックス（01_, 02_）で順序を管理。UIで並び替えると番号が自動で振り直される。フロントマターのYAMLにメタデータを含む。

ローカル vs クラウドの保存フロー

「保存」と「コミット」は別の操作です。保存は作業を失わないためのこまめな操作、コミットは意味のある区切りをGitの履歴として残す操作です。ローカルとクラウドで内部の仕組みが変わりますが、UIは共通です。

ローカル版 next dev で起動
1
自動保存（800ms debounce）
入力が止まって800ms後に fs.writeFile で .md / _course.json を書き込む
2
コミットボタン
コミットメッセージ入力 → サーバーで git add + git commit 実行
3
GitHub へ push（手動）
別のトレーナーと共有するとき、通常の git push で共有
クラウド版 Vercel にデプロイ
1
自動保存（800ms debounce）
入力が止まって800ms後に Neon DB の drafts テーブルへ UPSERT
2
コミットボタン（共通UI）
コミットメッセージ入力 → drafts を読んで GitHub API でリポジトリに書き込み
3
Vercel が自動再デプロイ
GitHub への commit を検知して Vercel が自動デプロイ（30〜60秒）

今回作成するテーブルは2つです。どちらもFK制約はあえて持たせず、lesson_id / entity_id はファイルのフロントマターIDと文字列で突き合わせます。

lesson_status_log ローカル・クラウド共通
カラム

今回の実装課題（3つ）

01

Vercel Blob — 画像保存

ローカル・クラウド共通

既存の画像APIルート（/api/images/upload、/api/images/promote）をVercel Blobに切り替える。ローカルから直接BlobへアップロードできるようAPIルートを改修する。

02

Vercel Neon — ステータス変更履歴

ローカル・クラウド共通

lesson_status_log テーブルを作成。UIでレッスンのステータスを変更したときにNeonへ記録。グローバルヘッダーに「アクティビティ履歴」ボタンを追加し、全レッスン横断の履歴ダイアログを実装する。

03

Vercel Neon + GitHub API — ドラフト保存とコミット

クラウド版のみ

drafts テーブルを作成。クラウドエディタとして動作するとき、編集内容をNeonへ自動保存し、「コミット」ボタンでGitHub APIを通じてリポジトリへ反映する。
