## 背景・新アイデアの発端

トレーニング資料の内容を作りこむにあたり、社内独自のプロセス・ルール・ツール環境などを考慮する必要がある。
そこで「**社内コンテキスト**」をデータベース化し、AI がレッスン草稿を生成する際に自動参照できるようにするアイデアが仲間とのディスカッションから生まれた。

### 具体的な課題感

- Python / GitHub の環境構築に社内独自のツールや申請が必要
- 特定プロジェクト向けの Git ブランチ戦略がある
- セキュリティ制約・アクセス権申請手順など気をつける情報が多い
- 社内資料は Docupedia や社内向け GitHub Pages にあるが、AI は直接アクセスできない
- NotebookLM は社内限定利用かつ API 非対応・Docupedia 直接参照不可（手動コピーが必要）

## 決まったこと

### 1. 社内コンテキストDBの用途・方針

| 項目 | 決定内容 |
|------|---------|
| **誰が使う** | AI がレッスン草稿生成時に自動参照する素材として使う |
| **誰が維持する** | 指定担当者が手動で登録・編集（自動同期なし） |
| **コンテンツ種類** | 業務プロセス、セキュリティ要件、環境構築手順（Python / GitHub 等）、Git ブランチ戦略、アクセス権申請手順 等 |
| **スコープ** | 原則 **全社共通**。プロジェクト固有情報はタグで表現（例: `xyz`） |

---

### 2. タグ設計

- **複数タグ付与可能**（1件のコンテキストに複数の側面を持たせる）
- **マスタ管理 ＋ 新規作成可**：既存タグから入力補完しつつ、新規タグをインラインで追加できる（GitHub Topics スタイル）
- **プロジェクト固有情報**は `xyz`（プロジェクト名）のようなタグを付けて区別。専用フィールドは不要

---

### 3. 鮮度管理

| フィールド | 必須/任意 | 内容 |
|------|------|------|
| `source_url` | **必須** | 原典資料の URL（Docupedia / GitHub Pages 等） |
| `source_last_updated_at` | 任意 | 原典資料の最終更新日（ページ記載の日付を手動入力） |

**アラート条件**: `source_last_updated_at` が 1年以上前、または未入力の場合にアラート表示

**フィルタ機能**: 管理ダイアログで「原典更新日が古いもの・未入力のもの」をフィルタ表示し、入力を促す

---

### 4. コンテンツ登録 UI

- **場所**: グローバルヘッダーに「社内コンテキスト」ボタンを追加 → ダイアログで CRUD

```
[DXトレーニング曼陀羅]  [社内コンテキスト]  [設定]
```

- **AI 要約ボタン（任意）**: 長い社内資料を貼り付けた後、「AI で要約」ボタンを押すとコンパクトに圧縮できる。短い内容はそのまま保存で構わない

### 5. レッスン草稿生成フロー（Pane 3 Agent ビュー）

#### UI 配置

```
[差分ビューボタン] [Agent ビューボタン]  ← Pane3 右上エリア
```

Agent ビューを開くと Cursor / Claude Code のような AI チャット欄が表示される。

#### 草稿生成スキルの動作フロー

```
1. トレーナーが「レッスン草稿作成」スキルを呼び出す
       ↓
2. スキルがレッスンタイトル・コース情報からコンテキストタグを自動選出
   → 「[環境構築, セキュリティ, xyz] で検索します。変更はありますか？」
       ↓
3. トレーナーがタグを確認・追加・削除して承認
       ↓
4. DB 検索 → 結果を提示
   「キーワード [環境構築, セキュリティ, xyz] で OO 件ヒットしました。
    これらの内容を盛り込みますか？
    社内コンテキストデータが足りない場合は、先に社内コンテキストダイアログから登録してください。」
       ↓
5. トレーナーが確認・承認 → レッスン草稿を生成
   - 取得したコンテキストアイテムは**全件まとめてプロンプトに渡す**（DB 検索は1回）
   - LLM が各アイテムの内容を判断し、レッスン内の**複数の適切な箇所に自動配置**する
   - 事前に「どこに入れるか」を人間が指定する必要はない
```

#### レッスン本文の構成パターン

```markdown
## [一般的なトピック解説]
...一般的な内容...

> **xyz ワンポイント**
> xyz プロジェクトにおけるブランチ戦略は...（xyz タグのコンテキストを反映）
```

### 7. `context_items` テーブル設計

```
┌─────────────────────────────────────────────────────────────┐
│                       context_items                          │
├──────────────────────┬──────────────┬───────────────────────┤
│ カラム               │ 型           │ 説明                   │
├──────────────────────┼──────────────┼───────────────────────┤
│ id                   │ SERIAL PK    │ 自動採番               │
│ title                │ TEXT NOT NULL│ コンテキスト項目のタイトル│
│ body                 │ TEXT NOT NULL│ 内容本文（手動入力 or  │
│                      │              │ AI 要約後のテキスト）   │
│ tags                 │ TEXT[]       │ 複数タグ（マスタ管理）  │
│ source_url           │ TEXT NOT NULL│ 原典資料 URL（必須）    │
│ source_last_updated_at│ DATE        │ 原典資料の最終更新日    │
│                      │              │（任意・手動入力）       │
│ created_by           │ TEXT         │ 登録者（OSユーザー名 or │
│                      │              │ 設定の表示名）          │
│ updated_by           │ TEXT         │ 最終更新者              │
│ created_at           │ TIMESTAMPTZ  │ DEFAULT now()          │
│ updated_at           │ TIMESTAMPTZ  │ DEFAULT now()          │
└──────────────────────┴──────────────┴───────────────────────┘
```

```sql
CREATE TABLE context_items (
  id                      SERIAL PRIMARY KEY,
  title                   TEXT NOT NULL,
  body                    TEXT NOT NULL,
  tags                    TEXT[] NOT NULL DEFAULT '{}',
  source_url              TEXT NOT NULL,
  source_last_updated_at  DATE,
  created_by              TEXT,
  updated_by              TEXT,
  created_at              TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at              TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_context_items_tags ON context_items USING GIN (tags);
```

**設計方針：**
- `tags` は PostgreSQL の `TEXT[]` 配列型で持つ。GIN インデックスでタグ検索を高速化
- `source_url` は必須。ツールからアクセスはできないが、担当者が手動で鮮度確認するためのアンカーとして機能
- `source_last_updated_at` は入力任意。NULL または 1年超でアラートフラグを UI 側で判定
- `created_by` / `updated_by` は FK なしの TEXT（`drafts.updated_by` と同じパターン）。ローカル版は `os.userInfo().username` を自動取得、クラウド版は WorkspaceSettings の表示名を使用。ユーザーテーブルは認証機能追加時に導入し、その際に FK マイグレーションで対応する

## 詰め切れていない事項

| 論点 | 内容 |
|------|------|
| **`body` のフォーマット** | プレーンテキストか Markdown か。長さ制限の要否 |
| **タグマスタの管理方法** | タグマスタをどのテーブル / ファイルで管理するか。初期タグセットの定義 |
| **GitHub OAuth** | 課題2のクラウド版で GitHub API を呼ぶ際の認証設計（NextAuth.js 等）は未設計 |
| **移行スクリプト** | 既存 `data/content.json` → `contents/` フォルダ構成への変換ツール |
| **Blob 内の画像 URL 参照** | markdown 本文内の画像参照を相対パスのままにするか Blob URL に変えるか |
| **ユーザーテーブル・認証** | 受講者への展開フェーズで導入。それまでは OS ユーザー名 / 表示名の TEXT で対応。複数トレーナー協業フェーズでは GitHub OAuth を先行検討 |
| **ゲーミフィケーション用テーブル** | `users` / `progress` / `trophies` 等は受講者展開フェーズで設計 |
