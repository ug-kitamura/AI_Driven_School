# Minutes Maid

月例会議の**音声文字起こし**から、部のメンバー全員が活用できる**図解形式議事録**を生成する。

## 対象読者の前提

- 部のメンバー23名（部長1名、DX5名、プロセス7名、Lab6名、トレーニング3名、庶務1名）
- **一番の対象は部長**: 品質管理・プロセスに詳しい
- 各メンバーはそれぞれが専門性を持っているが、お互いの業務詳細はよく知らない
- AIやプログラミングは非常に詳しい人が少数いる反面、全く分からない人が大半を占める
- 新しいことに積極的でないメンバーも何人かいる

→ **シンプル・明快・行動につながる** 議事録を目指す。絵文字・技術用語・略語は使わない。

---

## 最高の議事録の定義

以下の3つを達成できる議事録を「最高の議事録」とする。生成後の評価にも使用する。

1. **認識の統一** ── 会議で決まったこと・アクションプラン（誰が・いつまでに・何をするか）がテキスト化されており、参加者全員の認識を揃えられる（勘違いや解釈のずれをなくす）
2. **議論の解像度の保存** ── 会議中の議論によって解像度が上がった内容（背景・理由・注意点など）が余さずテキストにまとまっており、メンバーがタスクに取り掛かる際の補助となる
3. **パーパスとの接続** ── この会議全体の内容が部のパーパスにどう貢献しているかが示されており、メンバーが理解・振り返りをする際の補助となる

---

## 文章スタイルガイド

HTML内のテキストを書く際は、以下のルールを全セクションに適用する。

| ルール | 内容 |
|--------|------|
| **言葉遣い** | ビジネス文書として丁寧かつ簡潔に。冗長な表現・体言止めの多用は避ける |
| **トーン** | 全体的に少し柔らかく、親しみやすいトーンで。硬すぎず、くだけすぎず |
| **メンバー名** | 必ず「〜さん」付きで統一する（例：北村さん）。呼び捨て禁止 |
| **創作禁止** | 音声データ・参照ソースに存在しない内容を補足・創作しない。不明な事実は「要確認」に留める |
| **パーパス貢献** | 情熱的かつ前向きなトーンで記述する。「〜に不可欠な取り組みです！」「〜の貴重な機会となります！」など積極的な言葉を使う |
| **締めの言葉** | 締めの言葉として「ありがとうございました！」を加える |

## ワークフロー

### Phase 1: 文字起こし読み込み

まず **[references/purpose.md](references/purpose.md)** を読み、部のパーパスと10の戦略を把握する。

次に **[references/name-corrections.md](references/name-corrections.md)** を読み、固有名詞の補正リストを把握する。文字起こしを読む際はこのリストを適用し、誤った表記を正しい表記に置き換えて解釈すること。

続いて文字起こしファイルを読む。ファイルの特定方法：

1. ユーザーがファイル名を指定した場合 → そのファイルを読む
2. 指定がない場合 → カレントディレクトリの `.vtt` ファイルを探す
   - 1つだけ見つかった → そのファイルを読む
   - 複数見つかった → ユーザーにどれを使うか確認する
   - 見つからない → ユーザーにファイルパスを尋ねる

**Teams VTT の読み方：**

```
WEBVTT

00:00:01.000 --> 00:00:05.000
<v 北村>議題の1つ目ですが...</v>
```

- `<v 名前>` タグが話者。アクションプランの担当者特定に活用する
- タイムスタンプは通常不要だが、発言の前後関係の把握に使う
- 同一話者の連続発言は1つの発言としてまとめて解釈してよい

**会議の日付の特定:**

VTT 内に日付が明示されている場合はそれを使う。明示されていない場合は、ファイル名（例：`2026-03-30_meeting.vtt`）や発言の文脈から推測する。それでも不明な場合はユーザーに確認する。

チェック項目：

- 会議の日付・参加者・開催時間
- 議題のリスト
- 各議題での発言者と発言内容
- 明示・暗黙のアクション指示
- パーパス・戦略への言及

### Phase 2: 情報を4つのバケツに分類する

| バケツ | 内容 |
|-------|------|
| **決定事項** | 会議で合意・決定した内容 |
| **アクションプラン** | 誰が・いつまでに・何をするか |
| **タスク詳細** | 議論で解像度が上がった内容（背景・理由・注意点） |
| **パーパス貢献** | 各アイテムが部のパーパスにどうつながるか |

**重要**: アクションプランに「担当者不明」「期限不明」があれば、明示的にフラグを立てる。

### Phase 3: Markdownドラフトの書き出しとユーザー確認

Phase 2 の分類結果をもとに、以下の構造で Markdown ファイルを作成し `output/minutes-YYYY-MM-DD.md` に保存する（`YYYY-MM-DD` は会議の日付）。

**Markdownの構造:**

```markdown
# 会議名 議事録ドラフト
日時: YYYY年MM月 / 参加: N名 / 欠席: 〇〇さん・〇〇さん / 議事録担当: 〇〇さん

## サマリー
- 決定事項: N件 / アクション: N件 / 要確認: N件

## 議題リスト
1. 議題タイトル（報告者）
2. ...

## 議題の詳細

### 1. 議題タイトル
報告者・担当:

> [概要] 活動の概要や背景の説明文

> [注意点] 課題や懸念点の説明文（該当する場合のみ）

> [決定] 決定事項の説明文（決定がある場合のみ）

### 2. ...

## アクションプラン

| タスク | 担当 | 期限 | 状態 | 関連議題 |
|--------|------|------|------|---------|
| タスクの説明 | 〇〇さん | 4月15日 | 確定 | 議題1 |
| タスクの説明 | 〇〇さん | 要確認 | 要確認 | 議題2 |

## パーパスへの貢献

### 議題タイトル
戦略: (N) 戦略名 / (N) 戦略名

パーパス貢献の説明文（情熱的・前向きなトーンで）

### ...

ありがとうございました！
```

**保存後、必ずユーザーに以下を伝えて待機する:**

> `output/minutes-YYYY-MM-DD.md` を保存しました。内容をご確認・修正いただき、問題なければ「OK」とお知らせください。HTML生成に進みます。

### Phase 4: HTML生成

ユーザーから OK をもらったら `output/minutes-YYYY-MM-DD.md` を読み込み、その内容をもとに HTML を生成する。

**[references/base.html](references/base.html)** をベースフレームとして使い、`{{PLACEHOLDER}}` の部分を実際のコンテンツで埋める。

品質の目標イメージは **[references/model-answer.html](references/model-answer.html)** を参照（`<body>` の構造のみ参照すること）。

**プレースホルダーの置換（必須）:**

**スタイルのインライン化**: `base.html` の `<link rel="stylesheet" href="style.css">` を、**[references/style.css](references/style.css)** の内容を `<style>` タグとしてインライン化したものに置き換える。外部CSSリンクは残さない。

**Markdownの記法とHTMLクラスの対応:**

| Markdown | HTMLクラス |
|----------|-----------|
| `> [概要]` | `detail-box` |
| `> [注意点]` | `detail-box caution` |
| `> [決定]` | `detail-box decided` |
| 状態が「確定」のアクション行 | `action-row` |
| 状態が「要確認」のアクション行 | `action-row needs-check` |

**セクション構成順序:**

1. ヘッダー（会議名・日付・参加者数・欠席者・議事録担当）
2. ① サマリーカード（決定N件・アクションN件・要確認N件） + 要確認注意書き
3. ② 議題リスト（全議題タイトルを年表スタイルで一覧。各タイトルは議題の詳細へリンク）
4. ③ 議題の詳細（各議題の `mb-8` ブロックに `id="topic-N"` を付与）
5. ④ アクションプラン（議題ラベルは `#topic-N` へリンク）
6. ⑤ パーパスへの貢献（戦略ブロックで分けず議題順のフラットリスト。議題タイトルは `#topic-N` へリンク。貢献する戦略項目を `label-item` バッジで表示。複数並べてよい）

**各ブロックのルール:**

- `purpose-item` のテキスト内で戦略名キーワード（例：Sunshine Management、Pouring Water、Leaf 等）は `<span class="purpose-highlight">` でハイライトする

**アイコン選択** → [references/icon-list.md](references/icon-list.md)

出力ファイルは `output/minutes-YYYY-MM-DD.html` に保存する（Markdown と同じ日付）。

### Phase 5: アクションプランの検証

生成後、以下を確認する：

- [ ] 全アクションに担当者が明記されている（不明なら「要確認」バッジ）
- [ ] 全アクションに期限が明記されている（不明なら「要確認」バッジ）
- [ ] 担当者名が文字起こし内の表記と一致している
- [ ] 類似アクションが重複していない

**詳細** → [references/action-plan-format.md](references/action-plan-format.md)

### Phase 6: パーパス貢献の検証

Phase 4 で生成した HTML を読み、各議題のパーパス貢献の記述が適切かを確認する。マッピング自体は Phase 2〜3 で完了しているため、ここでは漏れ・誤りの検証に専念する。

**詳細** → [references/purpose-mapping.md](references/purpose-mapping.md)

### Phase 7: 品質評価

**[references/evaluate.md](references/evaluate.md)** の指示に従い、生成した HTML を評価して品質評価レポートを出力する。

## 品質チェックリスト

> このチェックリストは Phase 5〜7 で順に確認する。手動確認は不要。

### 必須

- [ ] アクションプランに「誰が・いつまでに・何をするか」が全て揃っている
- [ ] 「要確認」項目が明示的にハイライトされている
- [ ] 議論の解像度（背景・理由・注意点）がタスク詳細に含まれている
- [ ] パーパス貢献セクションがある

### 文章スタイル

- [ ] メンバー名が全て「〜さん」付きになっている
- [ ] 音声データにない情報を創作していない
- [ ] パーパス貢献の文章が前向き・情熱的なトーンになっている
- [ ] 末尾に「ありがとうございました！」が入っている

### デザイン

- [ ] Lucide iconのみ使用（絵文字禁止）。アイコン名は `references/icon-list.md` から選ぶ
- [ ] 「要確認」は赤系の色でハイライト
- [ ] スマホでも読みやすい（レスポンシブ）
- [ ] 専門用語・略語を使わない（初出は説明を添える）
