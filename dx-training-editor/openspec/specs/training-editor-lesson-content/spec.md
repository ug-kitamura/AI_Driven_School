# training-editor-lesson-content Specification

## Purpose

## Requirements
### Requirement: レッスンコンテンツはフロントマターと本文に分離する

各レッスンの `content` は、YAML フロントマター（`---` で囲む）と Markdown 本文から構成されなければならない（SHALL）。ワークスペース UI の編集モードでは `content` 文字列全体（フロントマターと本文）を表示・編集しなければならない（SHALL）。保存時、エディタは `applyLessonContentEdit` により `content` を更新し、先頭に有効なフロントマターがある場合はパース結果を `Lesson` フィールドに同期しなければならない（SHALL）。メタ編集モーダル保存時は、正規化されたメタと現在の本文から `content` を再生成しなければならない（SHALL）。

#### Scenario: 編集モードに content 全文が表示される

- **WHEN** ユーザーが Pane3 の編集モードを開く
- **THEN** エディタ領域に `---` で始まるフロントマターと Markdown 本文の両方が表示される

#### Scenario: 全文編集後に content が保存される

- **WHEN** ユーザーが編集モードで `content` を変更する
- **THEN** ワークスペース状態の `content` が編集後の文字列と一致する
- **AND** 先頭が `---` で始まる有効なドキュメントの場合、フロントマターから解釈可能な `Lesson` フィールドが同期される

#### Scenario: メタモーダル保存は本文を保持して FM を再生成する

- **WHEN** ユーザーがメタモーダルでメタのみを保存する
- **THEN** 本文は変更されず `content` のフロントマターのみ正規化される

### Requirement: フロントマターの正規化と親による上書き

読み込み時およびメタ保存時、エディタはフロントマターを正規化しなければならない（SHALL）。`series` および `course` は、当該レッスンの親シリーズ名・親コース名（ワークスペースのツリー）で上書きしなければならない（SHALL）。フロントマターの値が親と異なる場合も、保存時は親の日本語名が優先されなければならない（MUST）。

#### Scenario: 親のシリーズ名でフロントマターが上書きされる

- **WHEN** フロントマターの `series` が親シリーズ名と異なるレッスンを読み込む
- **THEN** 正規化後のメタの `series` は親シリーズの日本語名と一致する

#### Scenario: 欠損フィールドに初期値が入る

- **WHEN** フロントマターに `tags` が無い、または `estimated_minutes` が不正である
- **THEN** `tags` は空配列、`estimated_minutes` は 0 に正規化される（範囲外の数値は 0–180 にクランプ）

#### Scenario: レガシーステータス draft の移行

- **WHEN** フロントマターまたは `Lesson.status` が `draft` である
- **THEN** 正規化後のステータスは `open` となる

### Requirement: レッスンメタ編集モーダル

Pane3 ヘッダー最右の操作から、レッスンメタ編集ダイアログを開けなければならない（SHALL）。ダイアログは次の項目を提供しなければならない（SHALL）: シリーズ（読取・1件・日本語名）; コース（読取・1件・日本語名）; レッスン（編集可・単一行）; ステータス（`open` / `in_progress` / `done` から選択・アイコン表示・既定 `open`）; 説明（編集可・単一行）; タグ（編集可・複数・各要素は小文字英字・数字・ハイフンのみ）; 推定所要時間（分）（0–180 の整数・既定 0）; 著者（編集可・単一行・既定空）。

#### Scenario: シリーズとコースは編集できない

- **WHEN** レッスンメタ編集ダイアログを開く
- **THEN** シリーズとコースは読取表示のみである
- **AND** 表示名はワークスペース上の親の日本語名である

#### Scenario: メタ保存で Lesson と content が更新される

- **WHEN** ユーザーがメタを編集して保存する
- **THEN** 対応する `Lesson` のフィールドが更新される
- **AND** `content` のフロントマターが正規化された値で再生成される

### Requirement: Pane3 ヘッダーとモード切替

Pane3 のツールバー（高さ `h-12`）は、左に現在レッスンの日本語名（`lesson` フィールド）を表示しなければならない（SHALL）。モード切替（編集モード・プレビュー・差分）は右寄せで表示しなければならない（SHALL）。レッスンメタ編集の入口はツールバー最右に配置しなければならない（SHALL）。「ファイルを開く」操作を Pane3 に表示してはならない（MUST NOT）。編集モードの表示ラベルは「編集モード」とし、「Markdown」という表記を用いてはならない（MUST NOT）。

#### Scenario: ヘッダーにレッスン名とメタ編集がある

- **WHEN** レッスンが選択され Pane3 が表示される
- **THEN** ツールバー左にレッスン名が表示される
- **AND** ツールバー最右からメタ編集を開ける

#### Scenario: モードタブは右寄せ

- **WHEN** Pane3 ツールバーを表示する
- **THEN** 編集モード・プレビュー・差分のタブ群はレッスン名の右側（右寄せ）に配置される

#### Scenario: ファイルを開くがない

- **WHEN** Pane3 ツールバーを表示する
- **THEN** 「ファイルを開く」ボタンは存在しない

### Requirement: 編集モードの行番号

編集モードでは、エディタの左に 1 始まりの行番号を表示しなければならない（SHALL）。行番号は薄いグレーで視覚的に控えめとし、縦スクロールは編集領域と一体で同期しなければならない（SHALL）。

#### Scenario: 行番号が編集領域と同期スクロールする

- **WHEN** ユーザーが編集モードで縦スクロールする
- **THEN** 行番号ガターも同じスクロール位置に追随する

### Requirement: 編集モードの Markdown シンタックスハイライト

編集モードのエディタは、ライト時は VS Code または Cursor で Markdown を編集する場合と同等のシンタックスハイライトを提供しなければならない（SHALL）。ダーク時は **Cursor 現在のエディタ**に近い配色としなければならない（SHALL）。少なくとも次の要素を色分けして識別可能にしなければならない（SHALL）: ATX 見出し（`#` 〜 `######`）; 強調（`**` / `*`）; インラインコードとフェンスコードブロック; リストマーカー; リンク; GFM タスクリスト（`- [ ]` / `- [x]`）; フロントマター内の YAML キーと値。

#### Scenario: 見出しとリストが色分けされる

- **WHEN** ユーザーが `## 見出し` と `- 項目` を含むレッスンを編集モードで開く
- **THEN** 見出し行とリスト行がプレーンテキスト1色のみでは表示されない

#### Scenario: フロントマターが色分けされる

- **WHEN** ユーザーが `---` で囲まれたフロントマターを含むレッスンを編集モードで開く
- **THEN** YAML のキーと値が区別可能に色分けされる

#### Scenario: ダークで Cursor 風のコントラスト

- **WHEN** テーマが `dark` である
- **AND** ユーザーが編集モードを表示する
- **THEN** 背景とシンタックス色のコントラストが Cursor のダークエディタと同程度に読める

### Requirement: 編集モードの表示折りたたみ

編集モードは表示のみの折りたたみを提供しなければならない（SHALL）。折りたたみはソース文字列を変更してはならない（MUST NOT）。次を折りたたみ可能にしなければならない（SHALL）: 行頭 ATX 見出し `#` から `######` まで各見出し; 文書先頭のフロントマターブロック（最初の `---` から対応する閉じ `---` まで）。見出しを折りたたんだとき、当該見出し行自体はエディタ上に表示し続けなければならない（SHALL）。子孫見出しおよび本文行のみを非表示にしなければならない（SHALL）。見出しの折り範囲は、VS Code の Markdown 見出しフォールドと同等とし、見出しレベル L に対し次行から「レベルが L 以下の次の見出し」の直前までを隠すこととする（SHALL）。折りたたみ操作はガター上の折りたたみアイコン（`▶` / `▼`）で行えなければならない（SHALL）。

#### Scenario: 見出しを折りたたむと見出し行だけ残る

- **WHEN** ユーザーが `## セクション` の直下に本文があるレッスンで当該見出しを折りたたむ
- **THEN** `## セクション` の行は表示されたままである
- **AND** その直下の本文行は表示されない

#### Scenario: 子見出しは親見出しの折りたたみに含まれる

- **WHEN** ユーザーが `## 親` の下に `### 子` と本文がある状態で `## 親` を折りたたむ
- **THEN** `### 子` の行およびその本文も表示されない

#### Scenario: フロントマターを折りたためる

- **WHEN** ユーザーが先頭にフロントマターを持つレッスンでフロントマターブロックを折りたたむ
- **THEN** フロントマター内部の YAML 行は表示されない
- **AND** ブロックの開始 `---` 行は表示されたままである

#### Scenario: 初回表示はすべて展開

- **WHEN** ユーザーがレッスンを選択して編集モードを表示する、または別レッスンに切り替えて編集モードを表示する
- **THEN** すべての見出しおよびフロントマターブロックは展開された状態である

### Requirement: Pane4 からの Markdown 挿入

編集モードが表示されているとき、Pane4 の画像挿入操作は、エディタの現在のカーソル（選択）位置に Markdown 文字列を挿入しなければならない（SHALL）。挿入する URL は data URL ではなく `images/<filename>` 形式の相対パスとしなければならない（SHALL）。`images/uploaded/`・`images/ai/`・`images/web/` を挿入 URL に用いてはならない（MUST NOT）。AI タブからの挿入では `alt` に生成 API が返した **短い説明** を用いなければならない（SHALL）。UP タブからの挿入では従来どおりファイル名等を alt に用いてよい（MAY）。選択範囲がある場合は当該範囲を画像 Markdown に置換し、選択がない場合はカーソル位置に挿入しなければならない（SHALL）。挿入後、カーソルは挿入文字列の直後に置かれなければならない（SHALL）。プレビューモード・差分モード表示中に挿入操作を行った場合、挿入は行われず、ユーザーに編集モードへの切替を促すフィードバックを表示してよい（MAY）。

#### Scenario: 編集モードで画像 Markdown が挿入される

- **WHEN** ユーザーが編集モードでレッスンを開き、Pane4 から画像を挿入する
- **THEN** カーソル位置または選択範囲に `![...](images/<filename>)` 形式の文字列が反映される
- **AND** エディタはフォーカスを維持する

#### Scenario: AI 挿入は短い alt を含む

- **WHEN** ユーザーが AI タブから alt `commit から push まで` の画像を挿入する
- **THEN** 挿入文字列は `![commit から push まで](images/…)` 形式である

#### Scenario: data URL では挿入しない

- **WHEN** ユーザーが Pane4 から画像を挿入する
- **THEN** 挿入文字列に `data:` で始まる URL は含まれない

#### Scenario: プレビュー中は挿入されない

- **WHEN** ユーザーがプレビューモードを表示している
- **AND** Pane4 から挿入操作を行う
- **THEN** エディタ content は変更されない

### Requirement: プレビューは本文を GFM で表示する

プレビューモードでは、フロントマターをレンダリングしてはならない（MUST NOT）。本文のみを `react-markdown` でレンダリングし、GitHub Flavored Markdown（タスクリスト等）を有効にしなければならない（SHALL）。Typography は読みやすい資料向けのスタイル（例: `prose`）を用いなければならない（SHALL）。

#### Scenario: プレビューにフロントマターが出ない

- **WHEN** ユーザーがプレビューモードを開く
- **THEN** 画面上に YAML フロントマターのキー・値行は表示されない
- **AND** 本文の見出し・段落が表示される

### Requirement: 差分モードは content 全文を表示する

差分モードでは、当該レッスンの `content` 文字列全体（フロントマターと本文）について、**現在のワークスペース状態（メモリ上の `lesson.content`）** を左側（変更後）相当、**git HEAD 上の正本** を右側（変更前）相当として unified diff を表示しなければならない（SHALL）。差分の取得は **ユーザーが差分タブを選択したとき** に行わなければならない（SHALL）。編集モード表示中にバックグラウンドで差分を更新してはならない（MUST NOT）。差分表示は読み取り専用でなければならない（SHALL）。

HEAD 上の正本は次の順序で解決しなければならない（SHALL）:

1. `src/{series}/{course}/{lesson}.md`（`series` / `course` / `lesson` は当該 `Lesson` の日本語名フィールド）を `git show HEAD:` で取得
2. (1) が存在しない場合、`data/content.json` から同一 `lessonId` の `content` を取得
3. (2) も存在しない場合、正本は空文字列とする（新規レッスン）

差分 API はクライアントから `lessonId`・`content`（現在全文）・パス解決用の `series` / `course` / `lesson` を受け取らなければならない（SHALL）。レスポンスは unified diff 文字列を含まなければならない（SHALL）。変更が無い場合は空 diff とし、UI は「差分なし」を表示しなければならない（SHALL）。

#### Scenario: フロントマター変更が差分に含まれる

- **WHEN** ユーザーがメタのみを変更して差分タブを開く
- **THEN** 差分出力にフロントマター行の変更が含まれる

#### Scenario: 差分タブ選択時にのみ diff を取得する

- **WHEN** ユーザーが編集モードで `content` を変更する
- **THEN** 差分 API は呼び出されない
- **WHEN** ユーザーが差分タブを選択する
- **THEN** 現在の `lesson.content` を用いて diff が取得・表示される

#### Scenario: HEAD に md が無い場合 content.json を正本とする

- **WHEN** `src/{series}/{course}/{lesson}.md` が HEAD に存在しない
- **AND** `data/content.json` に同一 `lessonId` のレッスンがある
- **THEN** 差分の正本は当該レッスンの `content` である

#### Scenario: 新規レッスンは全行追加 diff

- **WHEN** HEAD 上に md も content.json 上の当該レッスンも存在しない
- **AND** ユーザーが差分タブを開く
- **THEN** 現在の `content` 全文が追加行として diff に表示される

#### Scenario: 変更が無い場合

- **WHEN** 現在の `content` が HEAD 正本と一致する
- **AND** ユーザーが差分タブを開く
- **THEN** UI は「差分なし」を表示する

### Requirement: 確認問題の記法

レッスン本文内の確認問題は、独自の directive 構文（例: `:::quiz`）を用いてはならない（MUST NOT）。確認問題は、見出し `### 確認問題` の直後に、GFM タスクリスト（`- [ ]` / `- [x]`）で記述しなければならない（SHALL）。見出し文言は `### 確認問題` に固定する（SHALL）。

#### Scenario: 確認問題は固定見出しとタスクリスト

- **WHEN** 作者が確認問題を本文に含める
- **THEN** `### 確認問題` が使われる
- **AND** 選択肢は `- [ ]` または `- [x]` のリストである

#### Scenario: プレビューでタスクリストが表示される

- **WHEN** 本文に `### 確認問題` とタスクリストがあるレッスンをプレビューする
- **THEN** 見出しとチェックボックス付きリストがレンダリングされる

### Requirement: レッスンステータスの語彙

レッスンのステータスは `open` / `in_progress` / `done` のいずれかでなければならない（SHALL）。 Pane3 メタモーダルおよび `Lesson.status` はこの語彙を用いなければならない（SHALL）。`draft` を新規データに保存してはならない（MUST NOT）。

#### Scenario: 新規レッスンの既定ステータス

- **WHEN** 新規レッスンが作成される
- **THEN** ステータスは `open` である

### Requirement: 骨子 AI 画像指示は HTML コメントで記述する

外部の骨子 AI スキルが生成するレッスン Markdown の **画像生成指示** は、HTML コメント `<!-- … -->` 内にプロンプト本文のみを記述しなければならない（SHALL）。`![…](images/…)` 形式の未生成プレースホルダを骨子段階で用いてはならない（MUST NOT）。コメントはプレビューで非表示となり、制作メモとして残してよい（MAY）。

#### Scenario: 骨子はコメントのみ

- **WHEN** 骨子 AI が画像生成指示を本文に含める
- **THEN** `<!-- プロンプト本文 -->` 形式で記述される
- **AND** `images/` への Markdown 画像リンクは含まれない

### Requirement: HTML コメント内カーソル時に AI タブプロンプトへ同期する

編集モードにおいて、CodeMirror のカーソル（または選択）が HTML コメント `<!-- … -->` **内** にある間、Pane4 AI タブのプロンプト入力欄は当該コメントの **内部テキスト**（`<!--` と `-->` を除く）で **常に** 更新されなければならない（SHALL）。カーソルがコメント外に移動した後も、プロンプト欄の内容は自動クリアしてはならない（MUST NOT）。AI タブが非表示でも、次回 AI タブ表示時にコメント内にカーソルがあれば同期されなければならない（SHALL）。

#### Scenario: コメント内に入るとプロンプトが更新される

- **WHEN** ユーザーが編集モードで `<!-- Git フロー図 -->` 内にカーソルを置く
- **THEN** AI タブのプロンプト欄に `Git フロー図` が表示される

#### Scenario: コメント外ではプロンプト欄を上書きしない

- **WHEN** ユーザーが AI タブでプロンプトを手編集した
- **AND** カーソルが HTML コメント外にある
- **THEN** 手編集内容はプロンプト欄に保持される

### Requirement: 編集モードで Ctrl とホイールによりフォントサイズを変更できる

Pane3 が **編集モード（raw）** のとき、編集エリア上で **Ctrl** を押しながらマウスホイールを回すと、フォントサイズを段階的に増減できなければならない（SHALL）。変更は **行番号ガター** を含め、本文と同じ比率でスケールしなければならない（SHALL）。操作はブラウザのページズームを発生させてはならない（MUST NOT）。変更後のサイズは `editorFontSizePx` として `dx-training-editor-settings` に永続化しなければならない（SHALL）。プレビュー・差分モードでは本操作を有効にしてはならない（MUST NOT）。許容範囲は実装で定義する min/max（例: 11–22 px）内に clamp しなければならない（SHALL）。

#### Scenario: Ctrl ホイールで拡大

- **WHEN** ユーザーが編集モードで Ctrl を押したままホイールを上に回す
- **THEN** エディタ本文のフォントサイズが 1 段階大きくなる
- **AND** 行番号のサイズも連動する

#### Scenario: プレビューではホイール変更しない

- **WHEN** ユーザーがプレビューモードを表示している
- **AND** Ctrl+ホイールを操作する
- **THEN** エディタフォントサイズは変更されない

#### Scenario: 変更が永続化される

- **WHEN** ユーザーが Ctrl+ホイールでフォントを 18 px に変更する
- **AND** ページを再読み込みする
- **THEN** 編集モードのフォントは 18 px から始まる
