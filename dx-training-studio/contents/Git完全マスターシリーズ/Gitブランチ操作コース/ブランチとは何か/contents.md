# ブランチとは何か

## 学習目標

このレッスンを終えると、次のことができるようになります。

- Git のブランチが何かを言葉で説明できる
- ブランチを使う理由を理解できる
- 代表的なブランチの種類（永続ブランチ・作業ブランチ）を区別できる

---

## ブランチのイメージ

Git のブランチは、**作業の「分岐点」** です。

木の幹から枝が伸びるように、メインの開発ラインから独立した作業ラインを作ることができます。枝の上で行った変更は、他の枝には影響しません。

<!-- 木の幹と枝のイラスト。幹が main ブランチ、枝が feature ブランチを表す図 -->

たとえば、次のような場面でブランチが役立ちます。

- 新機能を開発しながら、本番のコードを壊したくない
- 複数人が同時に別々の機能を開発したい
- 緊急のバグ修正を、開発中の機能と混ぜずに行いたい

---

## ブランチを使わないと何が起きる？

ブランチなしで全員が同じファイルを直接編集すると、次の問題が起きます。

- 他の人の変更と自分の変更が混ざってしまう
- 動作確認中の未完成コードが本番に混入する
- どの変更がいつ誰によって行われたか追跡しにくくなる

ブランチを使うことで、**それぞれの作業を安全に分離** できます。

---

## ブランチの種類

ブランチは大きく **永続ブランチ** と **作業ブランチ** に分けられます。

### 永続ブランチ

プロジェクトの間、ずっと存在し続けるブランチです。

| ブランチ名 | 役割 |
|---|---|
| `main` | 本番相当。常にデプロイ可能な状態を保つ |
| `develop` | 次リリース向けの統合ブランチ |

### 作業ブランチ

特定の目的のために一時的に作成し、作業完了後に削除するブランチです。

| 種別 | 用途 | 作成元 | 反映先 |
|---|---|---|---|
| `feature/*` | 機能開発 | `develop` | `develop` |
| `release/*` | リリース準備 | `develop` | `main` / `develop` |
| `hotfix/*` | 本番障害対応 | `main` | `main` / `develop` |

<!-- ブランチの流れ図。main / develop / feature / hotfix の関係を矢印で示す図 -->

## ブランチの基本的な流れ

1. 元になるブランチから新しいブランチを作成する
2. 新しいブランチ上で作業・コミットを行う
3. 作業が完了したら、元のブランチに変更を取り込む（マージ）
4. 不要になった作業ブランチを削除する

```bash
# develop ブランチから feature ブランチを作成して切り替える
git switch -c feature/my-new-feature develop

# 作業・コミット
git add .
git commit -m "新機能の実装"

# develop に戻る
git switch develop
```

## 社内での運用例

> **NMS ワンポイント**
>
> 社内プロジェクト NMS では、Git Flow をベースにしたブランチ運用ルールを採用しています。
>
> - ブランチ名には **チケット ID** を含める（例: `feature/NMS-1234-login-api`）
> - すべての変更は **Pull Request（PR）経由** で取り込む。直接 push は禁止
> - PR は CI（lint / unit test / build）が Green になるまでマージ不可
> - マージ後は作業ブランチを必ず削除する
>
> 詳細は [NMS Git ブランチ運用ルール](https://docupedia.example.local/new-major-system/git-branch-strategy) を参照してください。

## まとめ

- ブランチは作業を**安全に分離**するための仕組み
- **永続ブランチ**（`main` / `develop`）と**作業ブランチ**（`feature` / `release` / `hotfix`）がある
- 作業ブランチは目的ごとに作成し、完了後は削除する
- 次のレッスンでは、実際にブランチを作成・切り替える操作を学びます
