# Gitの三大エリア

## 学習目標

このレッスンを終えると、以下ができるようになります。

- Git の「作業ツリー」「ステージングエリア」「リポジトリ」の役割を説明できる
- 3 つのエリア間でファイルがどのように移動するかを理解できる
- 実際の操作ツール（SourceTree / VS Code / コマンドライン）でエリアの状態を確認できる

---

## Git の三大エリアとは

Git でファイルを管理するとき、変更は次の **3 つのエリア** を順番に通過します。

```
作業ツリー  ──(git add)──▶  ステージングエリア  ──(git commit)──▶  リポジトリ
```

## 各エリアの役割

### 1. 作業ツリー（Working Tree）

実際にファイルを編集する場所です。VS Code などのエディタで開いているフォルダがそのまま作業ツリーにあたります。

- ファイルを新規作成・編集・削除した変更は、まずここに反映される
- この時点ではまだ Git に「記録」されていない

> 💡 **ツールセットアップが済んでいない場合**  
> VS Code・Git・SourceTree のインストールには社内申請が必要です。  
> 研修開始の **1 週間前** までに申請を出しておきましょう。  
> 詳細は社内ドキュメント「[DX研修・日常開発向け 最低限ツールセットアップ手順](https://intranet.example.com/blog/dev-tools/setup-guide)」を参照してください。

### 2. ステージングエリア（Staging Area）

コミットする変更を選んで置いておく「待合室」です。`git add` コマンドで作業ツリーの変更をここへ移します。

- 複数ファイルを編集していても、**コミットしたいものだけを選んで** ステージングできる
- 「まだコミットしたくないファイル」はここに含めないことで、意図したまとまりでコミットが作れる

```bash
# ファイルをステージングエリアに追加する
git add ファイル名

# 変更のあるファイルをすべて追加する
git add .
```

### 3. リポジトリ（Repository）

確定した変更（コミット）が永続的に保存される場所です。`git commit` でステージングエリアの内容をここへ記録します。

- 一度コミットされた内容は履歴として残り、いつでも過去の状態に戻れる
- `.git` という隠しフォルダの中に保存されている

```bash
# ステージングエリアの内容をリポジトリに記録する
git commit -m "コミットメッセージ"
```

---

## エリア間のファイルの流れ（まとめ）

| 操作 | 移動元 | 移動先 |
|---|---|---|
| `git add` | 作業ツリー | ステージングエリア |
| `git commit` | ステージングエリア | リポジトリ |
| `git checkout` / `git restore` | リポジトリ / ステージング | 作業ツリー |

## ツールでエリアの状態を確認する

### コマンドラインの場合

```bash
# 現在の各エリアの状態を確認する
git status
```

### SourceTree の場合

- 「ファイルステータス」ビューで「作業ツリーのファイル」と「ステージングされたファイル」が視覚的に分かれて表示されます

## NMS プロジェクトでの活用例

> **NMS ワンポイント**  
> NMS プロジェクトでは Git Flow をベースにしたブランチ運用を採用しています。  
> 三大エリアの操作に慣れたら、以下のポイントも意識してみましょう。
>
> - **コミットメッセージ** には必ずチケット ID を付与する  
>   例: `NMS-1234: ログイン API のバリデーション追加`
> - **WIP コミット**（作業途中のコミット）は `develop` / `main` に残さない。PR 前に squash または fixup で整理する
> - すべての変更は **Pull Request 経由** で取り込む。ローカル merge のみでの共有は禁止
>
> 詳細は「[NMS Git ブランチ運用ルール](https://docupedia.example.local/new-major-system/git-branch-strategy)」を参照してください。
