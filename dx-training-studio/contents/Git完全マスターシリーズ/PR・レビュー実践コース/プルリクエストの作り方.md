# プルリクエストの作り方

## 学習目標

このレッスンを終えると、次のことができるようになります。

- フィーチャーブランチを作成してコミットをまとめる
- GitHub にブランチをプッシュして PR を作成する
- タイトル・説明文・レビュアーを適切に設定する
- PR がマージされるまでの流れを説明できる

---

## PR（プルリクエスト）とは

PR（Pull Request）は、自分が加えた変更をチームに共有し、レビューしてもらうための仕組みです。直接 `main` ブランチに変更を加えるのではなく、**専用のブランチで作業 → PR を作成 → レビュー → マージ** という流れをとることで、コードの品質を保ちつつチーム全体で変更内容を把握できます。

## 手順

### ステップ 1: フィーチャーブランチを作成する

`main`（または `develop`）ブランチから作業ブランチを切ります。

```bash
# develop ブランチを最新化
git switch develop
git pull origin develop

# フィーチャーブランチを作成して移動
git switch -c feature/1234-login-api
```

> **NMS ワンポイント**
> NMS プロジェクトでは、ブランチ名は **`feature/<チケットID>-<短い英語スラッグ>`** の形式で命名します。
> 例: `feature/NMS-1234-login-api`
> ブランチからブランチを切ること（例: `feature/A` から `feature/B`）は原則禁止です。詳細は [Git ブランチ戦略](https://docupedia.example.local/new-major-system/git-branch-strategy) を参照してください。

---

### ステップ 2: 変更を加えてコミットする

作業ブランチ上でファイルを編集し、コミットします。

```bash
# ファイルを編集後、ステージングしてコミット
git add .
git commit -m "NMS-1234: ログイン API のバリデーション追加"
```

> **NMS ワンポイント**
> コミットメッセージの先頭には **チケット ID** を付与します。
> 例: `NMS-1234: ログイン API のバリデーション追加`
> WIP コミットは PR を作成する前に `git rebase -i` で squash または fixup してまとめておきましょう。

---

### ステップ 3: GitHub にプッシュする

ローカルのブランチをリモートにプッシュします。

```bash
git push origin feature/1234-login-api
```

プッシュが成功すると、GitHub のリポジトリページに「Compare & pull request」ボタンが表示されます。

![GitHubリポジトリページの「Compare & pull request」ボタン表示モック](images/github-compare-pull-request-button.png)

---

### ステップ 4: PR を作成する

1. GitHub のリポジトリページで **「Compare & pull request」** をクリックします
2. **タイトル** にチケット ID と変更内容を簡潔に記述します
   - 例: `NMS-1234: ログイン API のバリデーション追加`
3. **説明文** に以下の内容を記載します
   - 変更の背景・目的
   - 変更内容の概要
   - 動作確認手順
   - 関連チケット・ドキュメントへのリンク
4. **Reviewers** にレビュアーを指定します
5. **「Create pull request」** をクリックして PR を作成します

<!-- プロンプト: GitHub の PR 作成画面（タイトル・説明文・Reviewers 欄）のスクリーンショットイメージ -->
![GitHub PR作成画面のブラウザUIモック（タイトル・説明文・Reviewers・Createボタン）](images/github-pr-creation-screen.png)

> **NMS ワンポイント**
> レビュアーは **1 名以上の Approver** を必ず指定します。自分自身を Approver に指定する Self-approve は禁止です。
> セキュリティに関わる変更（認証・認可・外部 API 連携など）は、通常のレビュアーに加えて **SecOps チーム** をレビュアーに追加してください。

---

### ステップ 5: レビューを受けてマージする

PR を作成したら、以下の流れでマージまで進めます。

1. **レビュアーからのコメントに対応する**
   - 修正が必要な場合はブランチ上でコミットを追加して再プッシュします
   - 対応済みのコメントは「Resolve conversation」で閉じましょう
2. **CI（自動テスト）がパスするのを確認する**
3. **Approve をもらう**
4. **マージする**

> **NMS ワンポイント**
> NMS プロジェクトでは以下のマージルールが定められています。
>
> - **CI が Green であること**（テスト・Lint すべてパス）
> - マージ方式は **Squash merge** を原則とする
> - マージは PR 作成者本人が行う
> - マージ後は **作業ブランチを削除する**（マージ実行者の責務）

```bash
# マージ後、ローカルのブランチを削除する場合
git switch develop
git pull origin develop
git branch -d feature/1234-login-api
```

## 確認ポイント

レッスンを終えたら、以下の項目をチェックしてみましょう。

- [ ] `main`/`develop` から正しい命名規則でブランチを切れたか
- [ ] コミットメッセージにチケット ID を含められたか
- [ ] PR のタイトル・説明文に必要な情報を記載できたか
- [ ] レビュアーを 1 名以上指定したか
- [ ] CI がパスしていることを確認してからマージしたか
- [ ] マージ後に作業ブランチを削除したか

---

## まとめ

| ステップ | 内容 |
|---------|------|
| 1 | フィーチャーブランチを作成 |
| 2 | 変更をコミット |
| 3 | GitHub にプッシュ |
| 4 | PR を作成（タイトル・説明・レビュアー設定） |
| 5 | レビュー対応 → CI パス → Approve → マージ → ブランチ削除 |

PR を通じた開発フローを習慣化することで、コードの品質向上とチーム内の知識共有が促進されます。次のレッスンでは、**レビューコメントの書き方と受け取り方** を学びます。
